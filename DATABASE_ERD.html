Legend

  • PK_(Entity name) (Datatype) = Primary Key (e.g., PK_users int)
  • FK1_(Entity name) (Datatype) = Foreign Key 1 (e.g., FK1_properties int)
  • FK2_(Entity name) (Datatype) = Foreign Key 2 (e.g., FK2_properties int)
  • UK_(Field name) (Datatype) = Unique Key (e.g., UK_email string)
  • Lines show relationships between tables
  • Cardinality: || = One, o{ = Many, o| = One (optional)

Relationship Summary

Core Relationships:

  • User → Properties: One-to-Many (user owns multiple properties)
  • User → Subscriptions: One-to-One (user has one subscription)
  • User → Tenant/Staff: One-to-One (user has one tenant or staff profile)
  • Subscription Plan → Subscriptions: One-to-Many (plan has many subscriptions)
  • Property → Units: One-to-Many (property has many units)
  • Property → Tenants: One-to-Many (property has many tenants)
  • Property → Staff: One-to-Many (property has many staff members)
  • Property → Inquiries: One-to-Many (property has many inquiries)
  • Tenant → Tenant Units: One-to-Many (tenant has many unit assignments)
  • Unit → Tenant Units: One-to-Many (unit has many tenant assignments)
  • Tenant → Bills: One-to-Many (tenant has many bills)
  • Unit → Bills: One-to-Many (unit has many bills)
  • Tenant Units → Bills: One-to-Many (tenant unit generates many bills)
  • Bill → Payments: One-to-Many (bill has many payments)
  • Tenant → Maintenance Requests: One-to-Many (tenant creates many requests)
  • Staff → Maintenance Requests: One-to-Many (staff assigned to many requests)
  • Property → Maintenance Requests: One-to-Many (property has many requests)
  • Unit → Maintenance Requests: One-to-Many (unit has many requests)
  • User → Tasks: One-to-Many (user creates/assigned to many tasks)
  • Property → Tasks: One-to-Many (property has many tasks)
  • Unit → Tasks: One-to-Many (unit has many tasks)
  • Staff → Tasks: One-to-Many (staff assigned to many tasks)
  • User → Inquiries: One-to-Many (user sends/receives inquiries)
  • Inquiry → Inquiry Messages: One-to-Many (inquiry has many messages)
  • Inquiry → Inquiry Attachments: One-to-Many (inquiry has many attachments)
  • Tenant → Chats: One-to-Many (tenant has many chats)
  • Property → Chats: One-to-Many (property has many chats)
  • Chat → Messages: One-to-Many (chat has many messages)
  • Property → Documents: One-to-Many (property has many documents)
  • User → Documents: One-to-Many (user uploads many documents)
  • Property → Announcements: One-to-Many (property has many announcements)
  • User → Announcements: One-to-Many (user publishes many announcements)
  • Property → Feedback: One-to-Many (property has many feedback entries)
  • User → Feedback: One-to-Many (user submits many feedback entries)
  • User → Notifications: One-to-Many (user receives many notifications)
  • Tenant → Notifications: One-to-Many (tenant receives many notifications)
  • User → Blacklisted Tokens: One-to-Many (user has many blacklisted tokens)
  • User → Payments: One-to-Many (user processes/verifies many payments)

System Overview:

Unified Architecture: This database schema supports both marketplace operations (property listings, inquiries, subscriptions) and property management operations (tenant management, billing, maintenance, staff, communication). The system uses a unified approach where core entities like users and properties serve both domains with appropriate relationships and fields.

Key Features:

  • Multi-tenant property management with sub-domain support
  • Complete billing and payment tracking system
  • Maintenance request workflow with staff assignment
  • Real-time communication via chat and messaging
  • Document management and announcements
  • Comprehensive notification system
  • Feedback and rating system
